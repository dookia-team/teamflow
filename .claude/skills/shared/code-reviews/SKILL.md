# Code Review Skill

## 목적
GitHub Pull Request에 대한 체계적이고 일관된 코드 리뷰를 수행하기 위한 가이드입니다.

---

## 코드 리뷰 프로세스

### 1. PR 정보 수집
```bash
# 열려있는 PR 확인
gh pr list

# 특정 PR 상세 정보
gh pr view {PR_NUMBER} --json number,title,author,baseRefName,headRefName,additions,deletions,files

# PR 작성자 확인
gh pr view {PR_NUMBER} --json author --jq '.author.login'
```

### 2. 변경 사항 분석
```bash
# PR diff 확인
gh pr diff {PR_NUMBER}

# 또는 로컬 브랜치 기반
git diff {BASE_BRANCH}...HEAD              # 전체 diff
git diff {BASE_BRANCH}...HEAD --stat       # 파일별 통계
git diff {BASE_BRANCH}...HEAD --name-only  # 변경 파일 목록
git log {BASE_BRANCH}..HEAD --oneline      # 커밋 목록
```

### 3. 심층 검토 수행

아래 체크리스트에 따라 검토합니다.

### 4. 리뷰 문서 작성

리뷰 결과를 템플릿에 맞춰 작성합니다.

### 5. PR에 리뷰 포스팅
```bash
# 코멘트로 포스팅
gh pr comment {PR_NUMBER} --body "$(cat review.md)"

# 승인 리뷰 (사용자 확인 필수)
gh pr review {PR_NUMBER} --approve --body "리뷰 내용"

# 변경 요청 리뷰
gh pr review {PR_NUMBER} --request-changes --body "리뷰 내용"
```

## 심층 검토 체크리스트

### React / Next.js 전용 검토

#### 컴포넌트 설계
- [ ] Server / Client Component 경계가 적절한가
- [ ] `'use client'`가 불필요하게 상위에 선언되지 않았는가
- [ ] 컴포넌트가 단일 책임 원칙을 따르는가
- [ ] Props 인터페이스가 명확하게 정의되었는가
- [ ] 컴포넌트 크기가 적절한가 (200줄 이하 권장)

#### 상태 관리
- [ ] 불필요한 `useState`/`useEffect` 사용이 없는가
- [ ] Server Component에서 처리 가능한 데이터를 클라이언트에서 fetch하고 있지 않은가
- [ ] 전역 상태가 과도하게 사용되지 않는가
- [ ] React Query의 queryKey가 적절하게 구성되었는가

#### 렌더링 성능
- [ ] 불필요한 리렌더링이 발생하지 않는가
- [ ] `useMemo`, `useCallback`이 적절히 사용되었는가 (과용 아닌지도 확인)
- [ ] 리스트 렌더링 시 `key`가 안정적인 값으로 설정되었는가
- [ ] 큰 목록에 가상화(virtualization)가 필요하지 않은가

#### Next.js 특화
- [ ] `metadata` / `generateMetadata`가 적절히 설정되었는가
- [ ] `loading.tsx`, `error.tsx`가 필요한 곳에 있는가
- [ ] `revalidate` / `cache` 전략이 적절한가
- [ ] Route Handler의 에러 처리가 적절한가
- [ ] 이미지에 `next/image`를 사용하고 있는가
- [ ] 링크에 `next/link`를 사용하고 있는가

### 공통 검토

#### 에러 핸들링
- [ ] API 호출 실패 시 적절한 에러 처리가 있는가
- [ ] null/undefined 참조 가능성이 없는가
- [ ] 사용자에게 적절한 에러 메시지를 보여주는가
- [ ] 엣지 케이스가 처리되었는가

#### 보안 (OWASP Top 10 기반)
- [ ] XSS: `dangerouslySetInnerHTML` 사용 시 sanitize 처리
- [ ] 사용자 입력값 검증이 적절한가
- [ ] 민감 정보가 클라이언트에 노출되지 않는가
- [ ] API 키, 토큰이 하드코딩되지 않았는가
- [ ] CSRF 토큰 관리가 적절한가
- [ ] httpOnly 쿠키 설정이 필요하지 않은가

#### 코드 품질
- [ ] 하드코딩된 값(매직 넘버, 매직 스트링)이 없는가
- [ ] 네이밍이 의미 있고 일관적인가
- [ ] 중복 코드가 없는가 (3번 이상 반복 시 추상화)
- [ ] TypeScript `any` 사용이 없는가
- [ ] import 순서가 컨벤션을 따르는가

#### FSD 아키텍처 준수
- [ ] 레이어 의존성 규칙 위반이 없는가 (하위→상위 import 금지)
- [ ] 같은 레이어 간 cross-import가 없는가
- [ ] Public API(index.ts)를 통해서만 외부 노출하는가
- [ ] 새 코드가 올바른 레이어에 배치되었는가

#### 테스트
- [ ] 핵심 비즈니스 로직에 테스트가 있는가
- [ ] 테스트가 의미 있는 케이스를 다루는가
- [ ] 엣지 케이스 테스트가 있는가

## 피드백 등급

### 우선순위 분류

| 등급 | 의미 | 머지 조건 |
|------|------|-----------|
| 🔴 **Must Fix** | 머지 불가 (버그, 보안, 데이터 손실) | 수정 후 재리뷰 |
| 🟡 **Should Fix** | 수정 권장 (설계, 하드코딩, 에러 핸들링) | 수정 또는 사유 설명 |
| 🟢 **Consider** | 개선 제안 (네이밍, 구조, 성능) | 머지 후 개선 가능 |

### 평가 등급

| 등급 | 조건 |
|------|------|
| 5/5 | Must Fix, Should Fix 없음. 즉시 머지 가능 |
| 4/5 | Should Fix 1~2건, Must Fix 없음. 피드백 확인 후 머지 |
| 3/5 | Must Fix 1건 또는 Should Fix 3건+. 수정 후 재검토 |
| 2/5 | Must Fix 다수. 대폭 수정 필요 |
| 1/5 | 재작성 필요 |

## 피드백 작성 규칙

### 원칙
1. **요약 먼저** — 바쁜 사람도 빠르게 파악 가능하게
2. **구체적 위치** — 파일명:라인번호 명시
3. **해결책 제시** — 문제만 지적하지 말고 개선안 코드 포함
4. **잘한 점도 언급** — 긍정적 피드백 포함
5. **기술적 근거** — 모든 피드백에 "왜"를 포함

### 코드 위치 표시
```
파일명:라인번호         예) src/features/auth/ui/LoginForm.tsx:44
파일명:시작-끝라인       예) src/shared/api/client.ts:12-25
```

### 피드백 포맷
```markdown
🔴 **Must Fix** — `src/features/auth/ui/LoginForm.tsx:44`

**문제**: 사용자 입력을 sanitize 없이 렌더링하고 있어 XSS 취약점이 있습니다.

**현재 코드**:
```tsx
<div dangerouslySetInnerHTML={{ __html: userInput }} />
```

**수정 제안**:
```tsx
import DOMPurify from 'dompurify';
<div dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(userInput) }} />
```
```

## PR 코멘트 포스팅 가이드

### 올바른 방법
```bash
# 방법 1: 변수로 읽어서 전달
CONTENT=$(cat review.md)
gh pr comment {PR_NUMBER} --body "$CONTENT"

# 방법 2: stdin 파이프
cat review.md | gh pr comment {PR_NUMBER} --body-file -

# 방법 3: Here Document
gh pr comment {PR_NUMBER} --body "$(cat <<'EOF'
[리뷰 내용]
EOF
)"
```

### 주의사항
- `--body-file 파일명` 대신 `--body "$CONTENT"` 사용 (파일 업로드 방지)
- PR 승인(`--approve`)은 **반드시 사용자 확인 후** 진행
- 중첩 코드 블록 시 외부: 백틱(```), 내부: 물결표(~~~) 사용
